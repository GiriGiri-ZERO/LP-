### 3.3 AIエンドポイント詳細

#### `POST /api/ai/generate`

**リクエストボディ:**
```json
{
  "doc_type": "lp",
  "product_context": {
    "product_name": "商品名",
    "product_description": "商品説明（500字以内）",
    "target_audience": "ターゲット顧客の説明",
    "unique_selling_points": ["USP1", "USP2", "USP3"],
    "price": "19,800円",
    "tone": "professional" // professional | friendly | luxury | urgent
  },
  "sections": ["hero", "features", "testimonials", "cta", "faq"]
}
```

**レスポンス:** `text/event-stream` (SSE)
```
data: {"type":"block_start","block_type":"hero"}
data: {"type":"delta","block_type":"hero","content":"..."}
data: {"type":"block_end","block_type":"hero","block_id":"uuid"}
data: {"type":"done"}
```

### 3.4 コードビュー同期仕様

エディタのHTML/CSSタブはサーバーへの保存なしにクライアント側でリアルタイム生成する。

```
[ビジュアル編集] ──→ blocks[] (Zustand store)
                           │
                     コード生成関数（クライアント）
                     generateHTML(blocks, theme)
                     generateCSS(blocks, theme)
                           │
                    [HTMLタブ] [CSSタブ]に表示
```

**HTMLタブで直接編集した場合のフロー:**
1. ユーザーがHTMLを編集 → デバウンス (500ms)
2. `PATCH /api/documents/:id/code` でサーバーに送信
3. サーバーがHTMLをパースしてブロック構造に変換を試みる
4. 変換可能な部分 → blocks[]を更新、プレビューに反映
5. 変換不能なカスタムHTML → `custom_html`フィールドとして保持し、プレビューに埋め込む

### 4.3 コピー生成ウィザード フロー

```
Step 1: ドキュメント種別選択
  → LP / セールスレター / Webページ / メール

Step 2: 商品・サービス情報入力
```

## 4.4 要素操作の詳細仕様（Canvaライク操作）

> **背景:** 現状の Canvas.tsx はブロック単位（行単位）の選択・編集のみ対応。
> テキスト要素・図形要素・画像要素を個別の「キャンバス上オブジェクト」として
> 自由移動・リサイズできる Canva ライクな操作感への拡張が必要。

### 4.4.1 テキスト操作の改善

**優先度: Must / 実装難易度: 中**

#### インタラクションモデル

```
[テキスト要素]
     │
     ├── シングルクリック ──→ 「選択」状態
     │                          ├── 移動ハンドル（要素上部にグリップアイコン）
     │                          ├── リサイズハンドル（8点コーナー＋辺）
     │                          └── フローティングツールバー表示
     │
     └── ダブルクリック  ──→ 「テキスト編集」モード
                                ├── テキストカーソル表示
                                ├── インライン入力（contenteditable または Tiptap）
                                └── 選択中はハンドル非表示
```

#### 状態遷移図

```
[未選択]
  │ シングルクリック
  ▼
[選択中] ──ドラッグ──→ [移動中]
  │  ↑                    │ 離す
  │  │ Esc / 外クリック    ▼
  │  └──────────────── [選択中]
  │ ダブルクリック
  ▼
[テキスト編集中]
  │ Esc / 外クリック
  ▼
[選択中]
```

#### 選択状態のUIレイアウト

```
 ○───────────────────○
 │  ┌──────────────┐ │   ← リサイズハンドル（●：コーナー、辺上の●：辺ハンドル）
 ●  │ テキスト内容  │ ●
 │  │               │ │
 ○───────────────────○
         ↑
   移動ハンドル（上部中央、ドラッグで移動）
```

#### 操作仕様一覧

| 操作 | 動作 | 備考 |
|---|---|---|
| シングルクリック | 要素を選択（ハンドル表示） | 他要素選択中は前要素を解除 |
| ダブルクリック | テキスト編集モード（カーソル） | contenteditable属性 or Tiptap |
| 選択状態でドラッグ | 要素を任意位置に移動 | `position: absolute` + `top/left`更新 |
| コーナーハンドルをドラッグ | 縦横同時リサイズ | `width/height`更新、アスペクト比ロック可 |
| 辺ハンドルをドラッグ | 一方向リサイズ（幅 or 高さのみ） | |
| Esc キー | 編集→選択→解除の順に状態を戻す | |
| Delete / Backspace | 選択中要素を削除（テキスト編集中は文字削除） | |
| 外部クリック | 選択解除 | キャンバス背景クリック |
| Shift + クリック | 複数要素を同時選択 | 将来対応（Could） |

### 4.4.2 左パネル → キャンバスへのドラッグ&ドロップ

**優先度: Must / 実装難易度: 中**

#### 概要

現状の SubPanel.tsx はクリックのみでブロック末尾に追加する実装。
D&D（ドラッグ＆ドロップ）でドロップ位置にブロックを挿入できるよう拡張する。

#### 操作フロー

```
[サブパネル（テキスト/図形/ブロック）]
         │
         │ ユーザーがアイテムをドラッグ開始
         ▼
[ドラッグゴースト表示（半透明プレビュー）]
         │
         │ キャンバスへ移動
         ▼
[キャンバスにドロップ位置インジケーター表示]
  ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─  ← 青い挿入ライン
  ┌──────────────────────┐
  │  HERO ブロック        │
  └──────────────────────┘
  ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─  ← ここにドロップ → ブロック挿入
  ┌──────────────────────┐
  │  FEATURES ブロック   │
  └──────────────────────┘
         │
         │ ドロップ
         ▼
[指定位置にブロックを挿入]
```

#### 技術方針

- @dnd-kit の `DragOverlay` と `useDroppable` を拡張して実装
- 既存の `DndContext`（ブロック並び替え用）に左パネルからの D&D を統合
- `draggable` データに `{ source: 'panel', block_type: BlockType }` を付与し
  既存のブロック間ドラッグと区別する
- ドロップ先はブロック間の `DropZoneLine` コンポーネントが受け取る

### 4.4.3 カラーパネルの要素適用

**優先度: Must / 実装難易度: 低**

#### 現状の問題

- SubPanel の `style` カテゴリは `updateTheme()` でテーマ全体を変更するのみ
- 選択中の要素（特定ブロックのテキスト色・背景色）への個別適用が未実装

#### 改善後の仕様

```
[カラーパネル]
  ├── テーマカラー変更（既存） ──→ document.theme を更新
  └── 選択要素の色変更（NEW）  ──→ selectedBlock.content.color を更新
```

| 色設定項目 | 適用先 | 対象ブロック種別 |
|---|---|---|
| テキスト色 | `content.color` または `content.text_color` | headline / body / hero / cta |
| 背景色 | `content.background_color` | hero / cta / shape |
| 塗りつぶし色 | `content.fill_color` | shape（図形ブロック） |
| 枠線色 | `content.border_color` | shape |

**適用ロジック:**
1. `selectedBlockId !== null` の場合 → 選択要素の色を更新（`updateBlock()` 呼び出し）
2. `selectedBlockId === null` の場合 → テーマカラーを更新（`updateTheme()` 呼び出し）
3. カラーピッカーの変更はリアルタイムで反映（デバウンスなし）

### 4.4.5 写真パネル（ローカル画像アップロード）

**優先度: Should / 実装難易度: 低**

#### 機能概要

```
[画像パネル]
  ┌────────────────────┐
  │  [+ 画像をアップロード]│  ← input[type=file] をトリガー
  ├────────────────────┤
  │  アップロード済み一覧  │
  │  ┌──┐ ┌──┐ ┌──┐   │
  │  │img│ │img│ │img│   │
  │  └──┘ └──┘ └──┘   │
  └────────────────────┘
  各画像はD&Dでキャンバスに配置
```

#### 処理フロー

1. ユーザーが「画像をアップロード」ボタンをクリック
2. `input[type="file" accept="image/*"]` のファイル選択ダイアログを表示
3. ファイル選択後 → `FileReader.readAsDataURL()` でBase64化（クライアント側）
4. Supabase Storage へアップロード → 公開URLを取得
5. 画像URLをパネルの「アップロード済み一覧」に追加
6. ユーザーが画像をD&Dまたはクリックでキャンバスに `image` ブロックを挿入
7. 挿入後は移動・リサイズ可能

#### ファイル制限

| 項目 | 制限値 |
|---|---|
| 対応フォーマット | JPEG / PNG / WebP / GIF / SVG |
| 最大ファイルサイズ | 10MB |
| 同時アップロード数 | 1ファイル（複数選択は将来対応） |

### 4.4.6 ブロック追加パネルのバグ修正

**優先度: Must / 実装難易度: 低**

#### 現状の問題

SubPanel.tsx の `blocks` カテゴリでクリックすると `addBlock(b.type)` が呼ばれるが、
Canvas.tsx 側でブロックが正しく表示されないケースがある（既知バグ）。

**推定原因:**
- `addBlock()` が `order_index` を末尾に設定するが、ソート後の位置計算が
  既存ブロックとの順序衝突を起こす可能性がある
- `document_id` が空文字列 `""` になるケース（ドキュメント未ロード時）

**修正方針:**
1. `addBlock()` 実行前に `document` の存在確認を行う
2. `order_index` の採番を `Math.max(...blocks.map(b => b.order_index)) + 1` に統一
3. SubPanel の blocks カテゴリは既存の `addBlock` を呼ぶだけでよく、
   コンポーネント側には実装変更不要

### 4.4.7 HTML/CSSタブのバグ修正と連動仕様

**優先度: Must / 実装難易度: 低〜中**

#### 現状の問題

CodeView.tsx の HTML/CSS タブを開くと以下の問題が発生：
- `generateHTML()` / `generateCSS()` 関数の呼び出し時に
  未定義プロパティへのアクセスが発生してエラーになる可能性
- `blocks` が空配列のとき空文字列が表示されるが、エラーが画面に残る
- `debounce()` の実装が `handleChange` を毎レンダーで再生成するため
  イベントハンドラが正しく動作しない（useCallback / useMemo 未使用）

#### 修正方針

| 問題 | 修正方法 |
|---|---|
| `generateHTML/CSS` でのランタイムエラー | try-catch でラップし、エラー時はエラーメッセージを表示 |
| debounce の毎レンダー再生成 | `useCallback` + `useRef` で debounce 関数をメモ化 |
| `doc` が `null` の場合の theme アクセス | `doc?.theme ?? {}` のガード（現状実装済みだが全箇所確認） |
| blocks 変化時の同期遅延 | `useEffect` の依存配列を正確に指定 |

#### リアルタイム連動仕様（強化）

```
[プレビュータブで要素を編集]
  │
  ▼
blocks[] / theme を Zustand store で更新
  │
  ▼
CodeView の useEffect が反応
  │
  ├── generateHTML(blocks, theme) → HTML表示を更新
  └── generateCSS(blocks, theme)  → CSS表示を更新

[HTML/CSSタブで直接編集]
  │ debounce 500ms
  ▼
PATCH /api/documents/:id/code
  │
  ▼
サーバーがパース → blocks[] を部分更新
  │
  ▼
Zustand store を更新 → プレビューに反映
```
