### 3.3 AIエンドポイント詳細

#### `POST /api/ai/generate`

**リクエストボディ:**
```json
{
  "doc_type": "lp",
  "product_context": {
    "product_name": "商品名",
    "product_description": "商品説明（500字以内）",
    "target_audience": "ターゲット顧客の説明",
    "unique_selling_points": ["USP1", "USP2", "USP3"],

```

**レスポンス:** `text/event-stream` (SSE)
```
data: {"type":"block_start","block_type":"hero"}
data: {"type":"delta","block_type":"hero","content":"..."}
data: {"type":"block_end","block_type":"hero","block_id":"uuid"}
data: {"type":"done"}
```

### 3.4 コードビュー同期仕様

エディタのHTML/CSSタブはサーバーへの保存なしにクライアント側でリアルタイム生成する。

```
[ビジュアル編集] ──→ blocks[] (Zustand store)
                           │
                     コード生成関数（クライアント）
                     generateHTML(blocks, theme)
                     generateCSS(blocks, theme)
                           │
                    [HTMLタブ] [CSSタブ]に表示
```

**HTMLタブで直接編集した場合のフロー:**
1. ユーザーがHTMLを編集 → デバウンス (500ms)
2. `PATCH /api/documents/:id/code` でサーバーに送信
3. サーバーがHTMLをパースしてブロック構造に変換を試みる
4. 変換可能な部分 → blocks[]を更新、プレビューに反映
5. 変換不能なカスタムHTML → `custom_html`フィールドとして保持し、プレビューに埋め込む


### 4.4.1 要素単位選択とフローティングツールバー（v1.2 新設計）

**優先度: Must / 実装難易度: 中**

#### Why（なぜこれが必要か）

現状はブロック全体が1つの選択単位で、色変更もブロック丸ごとにしか適用できない。
Canvaのように「見出しテキストをクリック → その文字色だけ変える」「ボタンをクリック → そのボタン色だけ変える」という
要素単位の操作ができないため、細かいデザイン調整ができない。

#### インタラクションモデル

```
[ブロック内の要素（テキスト/図形）]
     │
     ├── シングルクリック ──→ 「要素選択」状態
     │                          └── フローティングツールバーが要素の直上に出現
     │                               ├── テキスト要素: 色・サイズ・太字・配置
     │                               └── 図形要素: 塗り色・枠線色・角丸
     │
     └── ダブルクリック  ──→ 「テキスト編集」モード（contentEditable）
                                └── Escで選択状態に戻る
```

#### ワイヤーフレーム

```
  ┌──────────────────────────────────────────┐
  │  HERO ブロック                            │
  │  ┌──[A]色 [18px] [B][I] [≡][≡][≡]──┐  ← フローティングツールバー
  │  └─────────────────────────────────────┘
  │  ┌──────────────────────────────┐        │
  │  │  10分でプロ品質のLPを作成   │  ← 選択中（青枠）
  │  └──────────────────────────────┘        │
  │  サブヘッドライン                          │
  │  [  今すぐ申し込む  ]                     │
  └──────────────────────────────────────────┘
```

#### 状態管理（Zustand Store 変更）

**新規追加する型**

```typescript
// src/types/index.ts に追加
interface ElementStyle {
  color?: string;           // 文字色
  backgroundColor?: string; // 背景色（図形の塗り色）
  borderColor?: string;     // 枠線色
  borderRadius?: number;    // 角丸（px）
  fontSize?: number;        // フォントサイズ（px）
  fontWeight?: 'normal' | 'bold';
  fontStyle?: 'normal' | 'italic';
  textAlign?: 'left' | 'center' | 'right';
}

interface SelectedElement {
  blockId: string;
  elementId: string;   // 例: "headline", "subheadline", "cta_button", "card[0]"
  elementType: 'text' | 'shape';
}
```

**BlockContent への追加**

全ブロックContentインターフェースに以下を追加:
```typescript
elementStyles?: Record<string, ElementStyle>;
```
`elementStyles["headline"]`, `elementStyles["cta_button"]` のようにキーで各要素のスタイルを管理する。
これにより既存フィールド（`background_color`, `text_color` 等）は後方互換を保ちながら、新規要素スタイルを追加できる。

**EditorStore への追加アクション**

| アクション | 引数 | 説明 |
|---|---|---|
| `setSelectedElement` | `SelectedElement \| null` | 要素選択状態をセット |
| `updateElementStyle` | `(blockId, elementId, style: Partial<ElementStyle>)` | 特定要素のスタイルを更新 |

**EditorState への追加フィールド**

#### D-1: ダブルクリック特定要素のみ編集（バグ修正）

**優先度: Must / 難易度: 低**

##### Why（なぜ必要か）

現状の `editingBlockId` はブロック全体の編集モードを管理しており、

| 廃止 | `editingBlockId` | `editingElement` に統合。段階的に削除する |

```typescript
// src/types/index.ts に追加
interface EditingElement {
  blockId: string;
  elementId: string;  // 例: "headline", "subheadline", "cta_button"

```

```
[要素外クリック]

```

#### D-2: 8点リサイズハンドルオーバーレイ

**優先度: Must / 難易度: 中**

##### Why（なぜ必要か）

テキスト要素・図形要素・画像要素の**全要素共通**で、シングルクリック後に Canva のように  
要素の周囲にリサイズハンドルが現れ、ドラッグ操作でサイズを直感的に調整できるようにする。  
現状の FloatingElementToolbar の数値入力と並行して動作する（数値入力の視覚的補完）。

##### 全要素共通のシングルクリック挙動

テキスト要素・図形要素・画像要素のいずれをシングルクリックしても、以下が共通で発生する：

1. 要素を選択状態にする（`selectedElement` にセット）
2. 8点の丸いリサイズハンドルをバウンディングボックスとして表示
3. 要素ボディをドラッグ → 要素を移動（`transform: translate`）

##### UIレイアウト（テキスト・図形・画像 共通）

> **注意:** 8点ハンドルの表示レイアウトはテキスト・図形・画像の全要素タイプで**共通**。  
> ハンドルの見た目（丸い点）は同一。ドラッグ時の**効果のみ**が要素タイプによって異なる。

```
 ┌─●─────────────────●─┐
 │  [A] [18px] [B] [I]  │  ← FloatingElementToolbar（要素上部）
 ●                      ●  ← 辺ハンドル（左右）
 │  要素内容             │
 ●                      ●
 └─●─────────────────●─┘
     ↑ コーナーハンドル（4隅）
```

ハンドルは `position: fixed` の `ResizeHandleOverlay` コンポーネントが  
`getBoundingClientRect()` でDOM位置を追跡して配置する。

##### ハンドルドラッグ動作（elementType 別）

ハンドルの位置とカーソルは全要素で共通。ドラッグ時の効果が `elementType` によって異なる：

| ハンドル位置 | カーソル | text（テキスト要素） | shape（図形要素） | image（画像要素） |
|---|---|---|---|---|
| NW / NE / SW / SE（コーナー） | `nwse-resize` / `nesw-resize` | `fontSize` を比例スケール変更 | `width / height` を変更（縦横比維持 or フリー） | `width / height` を変更（縦横比維持 or フリー） |
| N / S（上下辺） | `ns-resize` | `fontSize` を増減（垂直方向のみ） | `height` のみ変更 | `height` のみ変更 |
| E / W（左右辺） | `ew-resize` | `elementStyles[id].width` を増減 | `width` のみ変更 | `width` のみ変更 |

> shape / image のコーナードラッグでは、`Shift` キー押下で縦横比維持ロックを解除（フリーサイズ）、  
> デフォルト（`Shift` なし）は縦横比を維持してリサイズする。

##### リサイズ計算ロジック（elementType 別）

```
onMouseDown（ハンドル）:
  → startMouseX/Y, startFontSize/startWidth/startHeight, elementType を記録
  → document.addEventListener('mousemove', onMouseMove)
  → document.addEventListener('mouseup', onMouseUp)

onMouseMove:
  → delta = currentXY - startMouseXY

  [text の場合]
    コーナー: newFontSize = clamp(startFontSize - deltaY * 0.3, 10, 120)
    上下辺:   newFontSize = clamp(startFontSize - deltaY * 0.3, 10, 120)
    左右辺:   newWidth    = clamp(startWidth + deltaX, 20, ∞)
    → updateElementStyle(blockId, elementId, { fontSize / width })

  [shape / image の場合]
    コーナー（Shiftなし）: 縦横比維持で width / height を算出
    コーナー（Shift押下）: newWidth / newHeight = start + delta（フリー）
    上辺 / 下辺: newHeight = startHeight ± deltaY
    左辺 / 右辺: newWidth  = startWidth  ± deltaX
    → updateElementStyle(blockId, elementId, { width, height })

onMouseUp:
  → removeEventListener
```

##### コンポーネント設計

```
src/components/editor/ResizeHandleOverlay.tsx  （新規）
  - Props: なし（store から selectedElement を読む）
  - selectedElement が null / editingElement が非null → 非表示
  - useEffect で selectedElement 変化時に getBoundingClientRect() を取得
  - 8個の handle div を position:fixed で配置（全要素タイプ共通レイアウト）
  - 各 handle に onMouseDown をバインド（elementType を参照して動作分岐）
  - createPortal(overlay, document.body) でレンダリング
```

##### 表示条件

| 条件 | 表示 |
|---|---|
| `selectedElement !== null` かつ `editingElement === null` | ✅ 表示（全要素タイプ共通） |
| `editingElement !== null`（テキスト編集中） | ❌ 非表示 |
| ブロックドラッグ中 | ❌ 非表示 |

#### D-3: 要素の位置移動（transform オフセット方式）

**優先度: Should / 難易度: 中**

##### Why（なぜ必要か）

テキスト要素・図形要素・画像要素の**全要素共通**で、選択した要素をドラッグしてブロック内の  
任意位置に移動できるようにする。  
`position: absolute` は LP縦スタック構造を壊すため採用しない。  
`transform: translate(offsetX, offsetY)` を使い、normal flow を維持したまま視覚的に移動する。

##### ダブルクリック挙動（elementType 別）

シングルクリックによる選択・移動は全要素共通だが、ダブルクリック時の動作は要素タイプによって異なる：

| 要素タイプ | ダブルクリック時の動作 |
|---|---|
| テキスト要素（text） | テキスト編集モードに移行（`contentEditable` / `editingElement` をセット） |
| 図形要素（shape） | 色変更パネルを開く（将来対応。現時点では何もしない） |
| 画像要素（image） | 画像差し替えダイアログを開く（将来対応。現時点では何もしない） |

> ダブルクリック時に `editingElement` がセットされるのはテキスト要素のみ。  
> 図形・画像要素ではダブルクリックしても `editingElement` は null のまま（リサイズハンドルは引き続き表示）。

##### データモデル変更

`ElementStyle` に位置オフセットと `height` を追加:

```typescript
// src/types/index.ts の ElementStyle に追加
interface ElementStyle {
  // 既存フィールド（色・フォント等）
  color?: string;
  fontSize?: number;
  // ... 既存フィールド省略 ...

  // Sprint D 追加
  offsetX?: number;  // px単位の水平オフセット（初期値: 0）
  offsetY?: number;  // px単位の垂直オフセット（初期値: 0）
  width?: number;    // 要素幅（px）。未指定時は natural width
  height?: number;   // 要素高さ（px）。shape / image に使用。text は不要
}
```

##### ブロックコンポーネントへのCSS適用

全要素タイプで共通の `transform` + `width` / `height` を適用:

```
style={{
  transform: `translate(${elementStyles?.[id]?.offsetX ?? 0}px, ${elementStyles?.[id]?.offsetY ?? 0}px)`,
  width:  elementStyles?.[id]?.width  ? `${elementStyles[id].width}px`  : undefined,
  height: elementStyles?.[id]?.height ? `${elementStyles[id].height}px` : undefined,
  position: 'relative',    // transformの基点を確保
  display: 'inline-block', // テキスト・インライン要素の幅が効くよう
}}
```

> `height` は shape / image 要素のみ実質的に使用する。text 要素はコンテンツ量で高さが決まるため原則設定しない。

##### 移動のインタラクション（全要素共通）

要素を「選択中（selectedElement）」かつ「編集中でない（editingElement === null）」状態でボディをドラッグ:

```
[要素本体の onMouseDown]（editingElement === null の時のみ）
  → startMouseX/Y, startOffsetX/Y を記録
  → document.addEventListener('mousemove', onMove)
  → document.addEventListener('mouseup', onUp)

[onMove]
  → dx = currentX - startMouseX
  → dy = currentY - startMouseY
  → updateElementStyle(blockId, elementId, {
      offsetX: startOffsetX + dx,
      offsetY: startOffsetY + dy,
    })

[onUp]
  → removeEventListener（確定）
```

> **注意:** 要素本体の onMouseDown で移動開始する場合、ダブルクリックと競合しないよう  
> `mousedown` と `dblclick` の区別を `event.detail`（クリック回数）で判定する。  
> `event.detail === 1` → 移動準備、`event.detail === 2` → ダブルクリック処理（移動キャンセル）。

##### 移動中のビジュアルフィードバック

- 移動中: カーソルが `grabbing` に変化
- 移動後: リサイズハンドルが新位置に追従（`getBoundingClientRect()` の再取得）
- ハンドルは `requestAnimationFrame` で追従（スムーズ更新）

##### リセット

フローティングツールバーに「位置リセット」ボタンを追加:  
`updateElementStyle(blockId, elementId, { offsetX: 0, offsetY: 0 })`

### 4.4.2 左パネル → キャンバスへのドラッグ&ドロップ

**優先度: Must / 実装難易度: 中**

#### 概要

現状の SubPanel.tsx はクリックのみでブロック末尾に追加する実装。
D&D（ドラッグ＆ドロップ）でドロップ位置にブロックを挿入できるよう拡張する。

#### 操作フロー

```
[サブパネル（テキスト/図形/ブロック）]
         │
         │ ユーザーがアイテムをドラッグ開始
         ▼
[ドラッグゴースト表示（半透明プレビュー）]
         │
         │ キャンバスへ移動
         ▼
[キャンバスにドロップ位置インジケーター表示]
  ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─  ← 青い挿入ライン
  ┌──────────────────────┐
  │  HERO ブロック        │
  └──────────────────────┘
  ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─  ← ここにドロップ → ブロック挿入
  ┌──────────────────────┐
  │  FEATURES ブロック   │
  └──────────────────────┘
         │
         │ ドロップ
         ▼
[指定位置にブロックを挿入]
```

#### 技術方針

- @dnd-kit の `DragOverlay` と `useDroppable` を拡張して実装
- 既存の `DndContext`（ブロック並び替え用）に左パネルからの D&D を統合

  既存のブロック間ドラッグと区別する
- ドロップ先はブロック間の `DropZoneLine` コンポーネントが受け取る

#### コードベース現状分析（SubPanel.tsx / Canvas.tsx）

| 項目 | 現状 | 必要な変更 |
|---|---|---|
| D&D ライブラリ | @dnd-kit 導入済み（`DndContext` / `useSortable`） | `useDraggable` をサブパネルアイテムに追加 |
| DragOverlay | Canvas.tsx に実装済み（ブロック並び替え用） | パネルからのドラッグにも対応するよう拡張 |
| ドロップゾーン | `DropZoneLine` コンポーネントが各ブロック間に存在 | `source: 'panel'` のデータを受け取るロジックを追加 |
| ドラッグデータ | `{ id: blockId }` のみ | `{ source: 'canvas' | 'panel', block_type?: BlockType }` に拡張 |

> `DndContext` を上位コンポーネント（エディタレイアウト）にリフトアップする必要がある。
> または Canvas.tsx に独自の `DndContext` を追加してネストさせる方法もあるが、
> @dnd-kit はネストされた `DndContext` をサポートするため後者も有効。

### 4.4.3 カラーパネルの要素適用

### 4.4.6 ブロック追加パネルのバグ修正

**優先度: Must / 実装難易度: 低**

#### 現状の問題

SubPanel.tsx の `blocks` カテゴリでクリックすると `addBlock(b.type)` が呼ばれるが、
Canvas.tsx 側でブロックが正しく表示されないケースがある（既知バグ）。

**推定原因:**
- `addBlock()` が `order_index` を末尾に設定するが、ソート後の位置計算が
  既存ブロックとの順序衝突を起こす可能性がある
- `document_id` が空文字列 `""` になるケース（ドキュメント未ロード時）

**修正方針:**
1. `addBlock()` 実行前に `document` の存在確認を行う
2. `order_index` の採番を `Math.max(...blocks.map(b => b.order_index)) + 1` に統一
3. SubPanel の blocks カテゴリは既存の `addBlock` を呼ぶだけでよく、
   コンポーネント側には実装変更不要

#### コードベース現状分析（store/editor.ts）

> **確認済み:** `addBlock()` 内で `order_index` の再採番は `state.blocks.forEach((b, i) => { b.order_index = i; })` で
> 毎回実行されており、順序衝突は発生しない実装になっている。
> バグの主因は `document_id: state.document?.id ?? ""` の空文字列問題と、
> `after_index` に `block.order_index`（ソート前の値）を渡している点にある可能性が高い。

**追加修正方針:**
- `addBlock()` 実行前に `if (!state.document) return;` ガードを追加
- Canvas.tsx の `onAddAfter={(type) => addBlock(type, block.order_index)}` を
  `addBlock(type, idx)` に変更（`idx` = 現在のソート済みインデックス）

### 4.4.7 HTML/CSSタブのバグ修正と連動仕様

**優先度: Must / 実装難易度: 低〜中**

CodeView.tsx の HTML/CSS タブを開くと以下の問題が発生：

| `doc` が `null` の場合の theme アクセス | `doc?.theme ?? {}` のガード（現状実装済みだが全箇所確認） |

```
  ▼

  │
  ▼

```

| 項目 | 現状 | 必要な変更 |

```typescript
    object_fit: 'cover',
    opacity: 1,
  } satisfies ImageContent;
```

### 5.3 自由配置モード共通インターフェース

`shape` / `image` ブロックはキャンバス上で自由配置（`position: absolute`）される。
将来的にテキストブロックも自由配置に対応できるよう、共通フラグを `Block` に追加する。

```typescript
// types/index.ts の Block インターフェースに追加フィールドを追加

interface Block {
  id: string
  document_id: string
  block_type: BlockType
  order_index: number
  content: BlockContent
  is_ai_generated: boolean
  // v1.1 追加フィールド
  is_free_positioned: boolean   // true: absolute配置、false: 縦スタック（既存動作）
  created_at: string
  updated_at: string
}
```

**モード判定ルール:**

| `is_free_positioned` | `block_type` | 描画モード |
|---|---|---|
| `false`（デフォルト） | hero / headline / body / cta など | 縦スタック（Sortable） |
| `true` | shape / image / （将来）text-free | `position: absolute` + x/y/width/height |

> **実装ノート:** Canvas.tsx では `block.is_free_positioned` を判定し、
> `true` のブロックは Sortable の外側（キャンバスの `position: relative` なオーバーレイ層）に
> `position: absolute` で描画する。Sortable リストとは独立したレイヤー構造とする。

```
キャンバス DOM 構造（推奨）:
┌─────────────────────────────────────────┐  position: relative
│  [Sortable Layer]  z-index: 0           │
│  ┌──────────────────────────────────┐   │
│  │ HEROブロック                     │   │
│  │ HEADLINEブロック                 │   │
│  └──────────────────────────────────┘   │
│                                         │
│  [Free Layer]  z-index: 10              │
│  ┌────────────┐  ┌────────────────┐     │
│  │ 矩形shape  │  │ image          │     │
│  └────────────┘  └────────────────┘     │
└─────────────────────────────────────────┘
```
